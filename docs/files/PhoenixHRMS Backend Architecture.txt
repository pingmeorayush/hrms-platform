1. Purpose
This document defines the backend architecture of PhoenixHRMS.
The backend is responsible for:
	•	Business logic
	•	Validation
	•	Security
	•	Data persistence
	•	Integrations
	•	Event processing
	•	Reporting
	•	Payroll processing
The backend shall act as the authoritative source of truth.

2. Backend Goals
The backend must provide:
	•	Scalability
	•	Maintainability
	•	Testability
	•	Security
	•	Extensibility
	•	Multi-tenancy
	•	High performance

4. Architectural Principles

BE-001
Business Logic Must Not Exist In Controllers
Controllers only coordinate requests.

BE-002
Business Logic Belongs To Services
Examples:
CreateEmployeeService
RunPayrollService
ApproveLeaveService

BE-003
Persistence Logic Belongs To Repositories
Repositories abstract database operations.

BE-004
Cross-Domain Communication Uses Events
Modules communicate via events.
Not direct dependencies.

BE-005
Everything Is Auditable
Every critical operation creates audit logs.

11. Repository Pattern
Repositories abstract database access.
Example:

Benefits:

	•	Flexibility
	•	Consistency

Example:

12. DTO Architecture

Conclusion
The PhoenixHRMS backend architecture is designed as a secure, scalable, event-driven modular monolith built on Laravel.
The architecture prioritizes maintainability, auditability, testability, and future scalability while remaining practical for a growing SaaS product.
All backend implementation must conform to the standards, patterns, and architectural principles defined in this document.
