Purpose
This document defines all user personas that interact with PhoenixHRMS.
The purpose of this document is to:
	•	Understand system users
	•	Define permissions
	•	Define workflows
	•	Define responsibilities
	•	Establish role-based access controls
	•	Drive dashboard design
	•	Drive UI navigation

Persona Classification
PhoenixHRMS supports the following personas:
System Level
	•	Super Administrator
	•	Tenant Administrator
HR Level
	•	HR Director
	•	HR Manager
	•	HR Executive
	•	Recruiter
	•	Payroll Officer
Business Level
	•	Department Head
	•	Manager
	•	Team Lead
Employee Level
	•	Employee

Persona 1: Super Administrator
Description
Highest privileged user within the platform.
Responsible for maintaining the entire SaaS environment.
Exists primarily in SaaS deployments.

Primary Goals
	•	Platform stability
	•	Security
	•	Tenant management
	•	Configuration management

Responsibilities
	•	Create organizations
	•	Configure subscriptions
	•	Manage global settings
	•	Manage integrations
	•	Manage security policies

Pain Points
	•	Unauthorized access
	•	Misconfigured tenants
	•	Security breaches
	•	Compliance risks

Key Metrics
	•	Platform uptime
	•	Active tenants
	•	Security incidents
	•	Subscription renewals

Dashboard Requirements
View:
	•	Total Organizations
	•	Total Employees
	•	Active Users
	•	Platform Health
	•	Subscription Status

Permissions
Full access.
Can:
	•	Create
	•	Read
	•	Update
	•	Delete
all records.

Persona 2: Tenant Administrator
Description
Represents organization owner.
Controls organization’s configuration.

Primary Goals
	•	Manage company settings
	•	Configure HR policies
	•	Manage users

Responsibilities
	•	Company setup
	•	User administration
	•	Department creation
	•	Policy management

Dashboard Requirements
	•	Employee Count
	•	Payroll Cost
	•	Attendance Rate
	•	Leave Statistics

Permissions
Full access within tenant.
Cannot access other organizations.

Persona 3: HR Director
Description
Strategic HR leader.
Focuses on workforce planning and organizational growth.

Goals
	•	Improve employee retention
	•	Improve hiring efficiency
	•	Reduce HR costs

Responsibilities
	•	Workforce planning
	•	Compensation strategy
	•	HR policy approval
	•	Performance oversight

KPIs
	•	Attrition Rate
	•	Retention Rate
	•	Hiring Success Rate
	•	Employee Satisfaction

Dashboard Widgets
	•	Attrition Trends
	•	Headcount Growth
	•	Payroll Cost Trends
	•	Recruitment Funnel

Permissions
Read:
	•	All employees
	•	All departments
Approve:
	•	Senior-level requests
	•	Compensation changes

Persona 11: Employee
Description
Standard user of the platform.
Represents majority of users.

Goals
	•	Access information
	•	Submit requests
	•	View payroll

Responsibilities
	•	Maintain profile
	•	Submit attendance
	•	Apply leave

Daily Activities
	•	Check attendance
	•	Submit leave requests
	•	Download payslips

Dashboard Widgets
	•	Attendance Summary
	•	Leave Balance
	•	Upcoming Holidays
	•	Payslip History

Permissions
View:
	•	Own profile
	•	Own attendance
	•	Own payroll
Create:
	•	Leave requests
Cannot:
	•	Access other employees

Data Access Matrix
Super Admin
All organizations

Tenant Admin
Own organization

HR Director
All employees within organization

HR Manager
All employees within organization

HR Executive
Limited HR scope

Recruiter
Recruitment data only

Payroll Officer
Payroll-related data only

Department Head
Department employees only

Manager
Direct reports only

Team Lead
Assigned team only

Employee
Self only

Dashboard Personalization Rules
Each user shall receive a personalized dashboard.
Examples:
Employee:
	•	Attendance
	•	Leave
	•	Payslips
Manager:
	•	Team Attendance
	•	Pending Approvals
HR Manager:
	•	Workforce Overview
	•	Recruitment Pipeline
Payroll Officer:
	•	Payroll Processing Queue
Tenant Admin:
	•	Organizational Overview

Notification Preferences
Every persona shall support:
	•	Email
	•	In-App
	•	SMS
	•	WhatsApp
Notification types depend on permissions.

Conclusion
These personas define the behavior, permissions, responsibilities, workflows, and data access patterns throughout PhoenixHRMS.
All future RBAC, dashboard design, workflow design, and security implementation must follow the definitions established within this document.
